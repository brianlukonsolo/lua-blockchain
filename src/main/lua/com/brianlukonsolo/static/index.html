Lesson 1

How To Read This Blockchain

This Learn section is not generic blockchain theory. It is a tour of this repository's node: a LuaJIT/OpenResty service, a SQLite-backed chain state, a browser wallet that signs transactions locally, and a native P2P plus HTTP sync layer that moves blocks and transactions between peers.

Read the sections in order if you want the full system story. Start in the Console if you want to operate the node while you learn. Every lesson pairs live node facts with a diagram so you can connect code, runtime behavior, and the exposed API.

Lesson 2

Architecture: From Browser To Persistent Chain State

The node is intentionally split into distinct responsibilities. OpenResty serves the frontend and API. The blockchain core validates and mutates state. SQLite holds durable chain, mempool, peer, and metadata tables. A native P2P daemon and optional gossip channel handle peer-side transport and discovery.

Why the split matters

Separating the HTTP surface, chain logic, transport daemons, and durable state makes the node easier to operate and reason about. It also makes the main bottlenecks visible: state reload cost, public snapshot size, and transport complexity.

Lesson 3

Transaction Flow: From Local Signature To Shared Mempool

In this frontend, wallets are generated in the browser with Web Crypto. The node never has to see your private key unless you deliberately use a server-side wallet route. The address is derived from the public key, and every outgoing transfer is signed over a canonical payload containing sender, recipient, amount, fee, nonce, note, and timestamp.

What the node enforces

The chain uses an account model. The node rejects invalid signatures, stale or skipped nonces, transactions that exceed the sender's pending balance, fees below the configured minimum, and notes that exceed the configured byte cap.

Lesson 4

Mining And Validation: How Blocks Become Canonical

Mining starts by selecting valid pending transfers, then appending a reward transaction for the miner. The block header includes the previous hash, proof, difficulty prefix, and cumulative work. The node retargets difficulty over a configurable time window rather than keeping a fixed proof target forever.

What “valid block” means here

The block must link to the previous hash, satisfy the proof target implied by its difficulty prefix, carry correct cumulative work, preserve account and nonce rules across all contained transfers, and leave the chain in a valid state when replayed from genesis.

Lesson 5

Consensus And Sync: Why Most Cumulative Work Wins

This node does not pick the longest chain blindly. It compares cumulative work. Peers advertise inventory and discovery data, exchange locators and headers, fetch missing blocks by hash, and only replace the local branch when the candidate chain proves more work.

Why headers-first matters

Headers are cheaper than full blocks. A node can inspect claimed work and tip continuity before it spends bandwidth on full block bodies. That makes sync more efficient and reduces pointless block transfer.

Lesson 6

Security And Operations: How The Node Protects Itself

The node uses several concentric control layers. Public HTTP is rate-limited. Administrative routes can require an admin token. Peer-facing routes can require a shared secret. Native P2P can run over TLS, and peer admission rules can enforce allowlists and per-IP or per-subnet caps. Backups snapshot the database and node identity material for recovery.

Operator mindset

Production hardening is not only about code. It also depends on secret rotation, restricted network paths, off-node backup retention, monitoring, and an honest understanding of which public interfaces are still expensive or revealing.

Glossary

Terms Used In This Node

Account model

Balances are tracked per address. Transactions mutate sender and recipient balances directly instead of spending discrete UTXOs.

Nonce

A monotonic counter per sender that prevents replay and enforces transaction ordering.

Difficulty prefix

The required leading-zero prefix for a valid block hash in this proof-of-work scheme.

Cumulative work

The total work represented by a chain. This node uses it as the fork-choice rule.

Headers-first sync

A sync strategy that compares headers and work before fetching full block bodies.

Peer directory

The peer-authenticated endpoint that returns advertised peers and node metadata for discovery.

Native P2P transport

The TCP-based peer channel that carries signed peer updates, block relay, and transaction propagation outside the ordinary public HTTP interface.

Operational backup

A point-in-time SQLite snapshot plus node identity and TLS materials used for recovery.
